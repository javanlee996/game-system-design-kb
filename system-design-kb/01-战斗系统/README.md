## 一、战斗系统的分类

### 1.1 按时间维度

| 类型 | 特征 | 代表游戏 |
|------|------|----------|
| **回合制** | 敌我轮流行动，可思考 | 宝可梦、星穹铁道、女神异闻录 |
| **即时制** | 实时输入，考验反应 | 鬼泣、只狼、黑神话 |
| **半即时制** | 时间流逝但可暂停/策略 | 博德之门3、FF7重制版 |
| **异步制** | 非同步交互，策略为主 | 炉石传说、自走棋 |

### 1.2 按空间维度

| 类型 | 特征 | 代表游戏 |
|------|------|----------|
| **锁定式** | 一对一或有限目标 | 魂系、怪物猎人 |
| **范围式** | AOE、方向判定 | 魔兽世界、暗黑破坏神 |
| **自由式** | 无硬锁定，动作游戏 | 鬼泣、猎天使魔女 |

### 1.3 按策略维度

| 类型 | 特征 | 代表游戏 |
|------|------|----------|
| **反应主导** | 闪避、格挡、连招 | 只狼、艾尔登法环 |
| **构建主导** | 配装、Build搭配 | 暗黑破坏神、POE |
| **资源主导** | 卡牌、CD、蓝量管理 | 炉石传说、LOL |

## 二、核心设计维度

### 2.1 输入维度

**玩家能做什么？**

```
基础动作：攻击、防御、闪避、技能
进阶动作：连招、取消、蓄力、切换
```

**设计原则**：
- 输入方式应该与游戏节奏匹配（回合制不需要复杂输入，动作游戏需要）
- 输入应该有"手感"（帧数据、输入缓冲、预输入）

**案例：只狼的格挡机制**
- 输入简单：只有一个格挡键
- 判定严格：完美格挡窗口短
- 反馈强烈：完美格挡有独特音效和火花
- 结果：简单输入产生高技巧天花板

### 2.2 决策维度

**玩家什么时候做决定？**

| 决策类型 | 说明 | 案例 |
|----------|------|------|
| **前瞻性决策** | 预判敌人动作 | 格斗游戏的立回 |
| **反应性决策** | 见招拆招 | 只狼的弹反 |
| **战略性决策** | 长周期资源分配 | RPG的技能CD管理 |
| **构建性决策** | 战前准备 | 暗黑破坏神的BD搭配 |

### 2.3 反馈维度

**玩家得到什么反馈？**

**即时反馈**（0-0.5秒）：
- 按键响应
- 伤害数字弹出
- 受击动画

**短期反馈**（0.5-5秒）：
- 连招完成
- 敌人硬直
- 技能CD转好

**中期反馈**（5秒-几分钟）：
- 战斗胜利
- 获得掉落
- 完成副本

**长期反馈**（跨战斗）：
- 角色成长
- 解锁新技能
- 装备提升

**设计原则**：反馈密度要与战斗时长匹配。一场5分钟的BOSS战需要密集的短期反馈，一场30秒的杂兵战则需要强烈的即时反馈。

### 2.4 风险-收益平衡

| 行为 | 风险 | 收益 | 案例 |
|------|------|------|------|
| 重攻击 | 前后摇大，易被打断 | 伤害高 | 怪物猎人的大剑 |
| 闪避 | 消耗体力/无敌帧有限 | 规避伤害+反击窗口 | 魂系的翻滚 |
| 蓄力 | 需要站桩 | 伤害翻倍+破防 | 原神的重击 |
| 贪刀 | 可能来不及闪避 | 多打一下输出 | 所有动作游戏 |

## 三、案例分析

### 3.1 回合制现代化：崩坏星穹铁道

👉 [详细分析见案例库](../99-案例库/星穹铁道战斗系统.md)

**核心启示**：通过"可视化行动条+韧性击破+战技点"三个机制，让传统回合制有了动作游戏的节奏感。

### 3.2 动作游戏的门槛控制：艾尔登法环

**设计特点**：
- 低门槛：召唤灵体、战技复读、魔法远程
- 高天花板：无伤、速杀、1级通关
- **关键设计**：难度选择不是菜单选项，而是游戏内选择（用不用骨灰、刷不刷等级）

### 3.3 异步策略：自走棋类

**设计特点**：
- 决策密度适中（每轮都有新决策）
- 运气成分创造话题性
- 构建乐趣（阵容搭配）

## 四、设计模板

### 模板1：战斗循环验证

```
1. 列出玩家的基础动作（攻击、闪避、技能等）
2. 画出这些动作的组合可能性
3. 问自己：
   - 最优解是什么？（是否存在唯一解）
   - 是否有"惩罚"行为？（玩家做了什么会被惩罚）
   - 是否有"奖励"行为？（玩家做了什么会被奖励）
4. 测试：一个新手和一个高手，战斗过程会有什么不同？
```

### 模板2：战斗难度曲线

```
1. 识别难度来源：
   □ 反应要求（需要多快的手速）
   □ 认知要求（需要同时关注多少信息）
   □ 知识要求（需要知道什么才能赢）
2. 设计难度调节阀：
   □ 数值调节（伤害、血量）
   □ 机制调节（是否使用某些机制）
   □ 信息调节（是否提前告知敌人动作）
3. 确保有"逃生通道"：难度过高时的退路
```

### 模板3：战斗-成长耦合检查

```
1. 战斗系统是否能让玩家感知到成长？
   - 数值成长（打得更快）
   - 机制成长（解锁新技能）
   - 技巧成长（玩家自己变强）
2. 成长是否会破坏战斗乐趣？
   - 过于简单导致无聊
   - 过于复杂导致焦虑
3. 是否有"展示成长"的舞台？
   - 木桩测试
   - 排行榜
   - 高难度挑战
```
